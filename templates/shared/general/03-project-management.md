# プロジェクト管理パターンガイド

## アジャイル開発手法

### スクラム フレームワーク

```markdown
## スプリント構成（2週間サイクル）

### Week 1

- 月曜日: スプリントプランニング
- 火〜金曜日: 開発作業 + デイリースクラム
- 金曜日: 中間レビュー

### Week 2

- 月〜木曜日: 開発作業 + デイリースクラム
- 金曜日: スプリントレビュー + レトロスペクティブ

## 主要イベント

### スプリントプランニング（4時間）

1. プロダクトバックログの確認
2. スプリントゴールの設定
3. スプリントバックログの作成
4. タスクの見積もり

### デイリースクラム（15分）

- 昨日やったこと
- 今日やること
- 障害・ブロッカー

### スプリントレビュー（2時間）

- デモンストレーション
- ステークホルダーフィードバック
- 完了定義の確認

### レトロスペクティブ（1時間）

- Keep: 続けること
- Problem: 問題点
- Try: 改善案
```

### ユーザーストーリー作成

```markdown
## ユーザーストーリーテンプレート

As a [role/persona]
I want [goal/desire]
So that [benefit/value]

## 例

As a registered user
I want to reset my password
So that I can regain access to my account if I forget it

## 受け入れ基準（Given-When-Then形式）

Given: 前提条件
When: 実行する操作
Then: 期待される結果

### 例

Given: ユーザーがログインページにいる
When: "パスワードを忘れた"リンクをクリックする
Then: パスワードリセットページが表示される

Given: 有効なメールアドレスを入力
When: "リセットメール送信"ボタンをクリック
Then: リセット用のメールが送信される
```

バグの簡潔な説明

1. 〇〇画面を開く
2. 〇〇ボタンをクリック
3. エラーが発生

**提案する解決策**
どのような実装を想定しているか

他に考えた解決策があるか

**追加コンテキスト**
その他の関連情報

// プロジェクトメトリクス型定義
interface ProjectMetrics {
  sprint: {
    number: number
    velocity: number
    commitment: number
    burndownData: BurndownPoint[]
  }
  quality: {
    testCoverage: number
    bugCount: number
    codeQualityScore: number
  }
  team: {
    memberCount: number
    availableHours: number
    utilization: number
  }
}

// メトリクス収集
const collectSprintMetrics = async (sprintId: string): Promise<ProjectMetrics> => {
  const [sprintData, qualityData, teamData] = await Promise.all([
    getSprintData(sprintId),
    getQualityMetrics(sprintId),
    getTeamMetrics(sprintId),
  ])

  return {
    sprint: sprintData,
    quality: qualityData,
    team: teamData,
  }
}

## リスク管理

### リスクアセスメント

```markdown
## リスク分析フレームワーク

### 技術的リスク

| リスク               | 確率 | 影響度 | 対策                     |
| -------------------- | ---- | ------ | ------------------------ |
| ライブラリの非互換性 | 中   | 高     | プロトタイプでの事前検証 |
| パフォーマンス問題   | 低   | 中     | 早期パフォーマンステスト |
| セキュリティ脆弱性   | 中   | 高     | 定期的なセキュリティ監査 |

### プロジェクトリスク

| リスク               | 確率 | 影響度 | 対策                       |
| -------------------- | ---- | ------ | -------------------------- |
| 要件変更             | 高   | 中     | アジャイル手法での対応     |
| リソース不足         | 中   | 高     | バッファの確保             |
| ステークホルダー離脱 | 低   | 高     | 定期的なコミュニケーション |

### 対応戦略

1. **回避**: リスクを発生させない
2. **軽減**: リスクの影響を最小化
3. **転嫁**: リスクを第三者に移転
4. **受容**: リスクを受け入れて対処
```

## プロジェクト振り返り

### レトロスペクティブ手法

```markdown
## KPT (Keep, Problem, Try)

### Keep (続けること)

- 効果的だったプラクティス
- チームがうまく機能した点
- 継続すべき行動

### Problem (問題点)

- 困難だった事柄
- 改善が必要な領域
- ブロッカーとなった要因

### Try (試すこと)

- 新しく試したいアプローチ
- 問題解決のためのアクション
- 実験的な取り組み

## 4L (Liked, Learned, Lacked, Longed for)

### Liked (良かったこと)

- 成功した取り組み
- 満足できた結果

### Learned (学んだこと)

- 新しい知識・スキル
- 得られた洞察

### Lacked (足りなかったこと)

- 不足していたリソース
- 改善の余地

### Longed for (あったらよかったもの)

- 必要だったツール
- 欲しかったサポート
```

## ツール活用

### プロジェクト管理ツール選択

```markdown
## ツール比較

### Jira + Confluence

- **適用**: 大規模チーム、複雑なプロジェクト
- **特徴**: 豊富な機能、詳細な設定可能
- **コスト**: 有料（ユーザーあたり）

### GitHub Projects

- **適用**: 開発中心のチーム
- **特徴**: Git との連携、シンプル
- **コスト**: 無料（GitHubプランに含まれる）

### Linear

- **適用**: スタートアップ、小〜中規模チーム
- **特徴**: 高速、直感的なUI
- **コスト**: 有料（ユーザーあたり）

### Notion

- **適用**: ドキュメント管理重視
- **特徴**: 柔軟なデータベース機能
- **コスト**: 一部無料、有料プランあり
```
